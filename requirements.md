# 项目需求书（总文档）

## 1. 项目背景与目标

### 1.1 项目背景

当前市场中的待办类工具大多只关注短期任务，而长期规划与短期执行之间缺乏自然连接。用户需要的是一个统一的个人规划系统，既能管理长期目标与阶段计划，又能管理日常任务与短期安排，并且能够看动态、看全局、看执行。

本项目目标是构建一个**个人长期规划与短期计划管理系统**，帮助个人用户把目标、计划、任务与执行串联成一个完整闭环。

### 1.2 项目目标

- 为用户提供一个统一的个人空间，用于管理长期目标、阶段计划、短期任务和备忘录。
- 首页固定由三个核心区域组成：信息流、甘特图、日历。
- 计划采用层级模型：`goal（目标）→ phase（阶段）→ weekly/daily（短期计划）`，任务挂载在计划下。
- 新增任务或计划后，系统根据日期字段自动分流（详见第 6 节决策表）。
- 支持任务层级结构：任务可以选择父任务作为子任务，也可以作为独立父任务。
- 支持管理员后台，用于管理用户与订阅，但不参与用户个人规划内容。
- MVP 阶段优先实现用户端核心闭环；复盘、真实支付等留 v2。

## 2. 目标用户与使用场景

### 2.1 目标用户

- 需要长期规划的个人用户（如学生、职场人士、自由职业者）。
- 需要将目标拆解为具体任务的用户。
- 需要统一看动态、看全局、看执行的用户。
- 不需要多人在同一个任务中协作的单用户场景。

### 2.2 典型使用场景

- 用户创建长期目标（goal），并拆解为多个阶段计划（phase）。
- 用户为阶段计划创建周计划/日计划（weekly/daily），并关联具体任务。
- 用户新增任务时来不及填写日期，内容进入备忘录等待补充。
- 用户在备忘录中补充日期后，Memo 被删除，内容回流到日历和甘特图。
- 管理员在后台查看用户注册与订阅状态，手动维护套餐有效期。

## 3. 用户模式与空间设计

### 3.1 用户模式

- 系统采用**单用户独立空间**模式。
- 每个用户拥有自己的私有空间，只能访问自己的数据。
- 不提供多用户协作、任务分配、团队权限等功能。

### 3.2 管理员后台

- 管理员后台独立于用户个人空间。
- 管理员不参与用户个人规划内容。
- MVP 后台范围：用户管理、订阅状态管理。
- 普通用户不能访问后台。

## 4. 首页结构

首页固定由三个核心区域组成；桌面端同屏展示，移动端 Tab 切换（见 `docs/routes.md`）。

### 4.1 信息流（动态区）

- 展示最近动态：新增、编辑、完成、归档。
- 每条记录包含：动作类型、关联内容类型（任务 / 计划 / 备忘录）、时间戳。
- 按时间倒序排列。

### 4.2 甘特图（全局区）

- 展示 goal、phase、任务的时间条及父子任务层级。
- 有 start 的条目均展示；若尚无 due/end，使用**虚拟截止日**（见 6.2）。
- 虚拟截止 = `start + 365 天`，每次渲染按当天重新计算；若该值早于今天则延长到今天。
- 用户填写确定截止日后，改用真实 due/end，虚拟规则停止。
- 虚拟截止的甘特条需有视觉区分（虚线或「预估截止」标签）。
- 支持时间范围选择。

### 4.3 日历（执行区）

- 展示有开始日期的任务和计划（仅有 start、尚无 due/end 的条目也显示）。
- 支持月 / 周 / 日视图，点击条目查看详情。

**首页设计原则**：信息流 = 看动态；甘特图 = 看全局；日历 = 看执行。

## 5. 核心功能清单

### 5.1 长期规划

- 用户可创建 type = `goal` 的长期目标。
- 目标下可创建 type = `phase` 的阶段计划（`parent_plan_id` 指向 goal）。
- 阶段计划可关联任务（Task.plan_id）。
- 长期规划页展示 goal → phase → 任务树。

### 5.2 短期计划

- 用户可创建 type = `weekly` 或 `daily` 的短期计划。
- 可选挂载到某个 phase（`parent_plan_id`），也可独立存在。
- 短期计划可关联任务，支持查看完成情况。

### 5.3 新增任务

- 支持字段：标题（必填）、描述、状态、优先级、开始日期、截止日期、所属计划、父任务。
- **日期约束**：禁止只填截止日期不填开始日期；若两者都填，due >= start。
- 新增后写入信息流；按第 6 节分流。

### 5.4 新增计划

- 支持字段：标题（必填）、描述、type（goal / phase / weekly / daily）、父计划、开始日期、结束日期、状态。
- **层级约束**：phase 的父计划必须是 goal；weekly/daily 的父计划可为空或 phase。
- **日期约束**：禁止只填结束日期不填开始日期；若两者都填，end >= start。

### 5.5 父任务与子任务

- 可选父任务；未选则为独立父任务。
- 层级最多 3 层，禁止循环引用。
- 父任务汇总规则（展示层推导）：
  - 全部子任务 done → 父任务显示 done。
  - 任一 in_progress，或部分 done → 父任务显示 in_progress。
  - 其余 → todo。

### 5.6 任务管理

- 支持查看、编辑、完成、延期、归档、删除。
- 编辑后：写 Feed；日期变更走分流；父任务变更更新层级；若有关联 Memo 则同步 title/description。

### 5.7 复盘记录（v2，不在 MVP）

- v2 功能：周/月/阶段复盘，关联 goal 或 phase，摘要写入信息流。
- MVP 不实现。

### 5.8 备忘录

- 承接**无任何日期**的任务或计划（独立 Memo 表，关联源记录）。
- 支持查看、编辑说明、补充日期后回流、删除/归档。
- 编辑任务/计划标题或描述时，**必须同步**关联 Memo。
- 补充日期满足回流条件后，**硬删除** Memo（不保留 Memo 历史）。

## 6. 新增与分流规则（核心业务逻辑）

所有分流逻辑集中在 `lib/content-router.ts`，禁止各视图各自实现。

### 6.1 新增/变更均写入信息流

- 新增、编辑、完成、归档任务/计划均生成 Feed 记录。

### 6.2 分流决策表

**任务**

| start_date | due_date | 日历 | 甘特图 | 备忘录 |
|------------|----------|------|--------|--------|
| 无 | 无 | ✗ | ✗ | ✓ |
| 有 | 无 | ✓ | ✓（虚拟截止） | ✗ |
| 有 | 有 | ✓ | ✓（真实 due） | ✗ |
| 无 | 有 | — | — | **禁止（校验失败）** |

**计划**

| start_date | end_date | 日历 | 甘特图 | 备忘录 |
|------------|----------|------|--------|--------|
| 无 | 无 | ✗ | ✗ | ✓ |
| 有 | 无 | ✓ | ✓（虚拟截止） | ✗ |
| 有 | 有 | ✓ | ✓（真实 end） | ✗ |
| 无 | 有 | — | — | **禁止（校验失败）** |

**甘特图虚拟截止日（仅有 start 时）**

```
effective_end = max(start_date + 365 天, today)
```

- 不写入数据库；每次加载甘特图时按当天重新计算（动态变化）。
- 用户填写 due_date / end_date 后，改用真实值，虚拟规则失效。

### 6.3 备忘录创建与删除

- 进入备忘录：任务/计划无任何日期 → 自动创建 Memo（linked_task_id 或 linked_plan_id）。
- 离开备忘录：补全 start（达到日历条件）→ **硬删除** Memo；条目进入日历，有 start 即进入甘特图（无 due/end 时用虚拟截止）。
- 再次清空全部日期 → 重新创建 Memo。

### 6.4 规则总结

- 任何变更 → Feed。
- 有 start → 日历 + 甘特图（甘特图截止为真实 due/end，或虚拟 effective_end）。
- 无任何日期 → 备忘录（Memo 表）。
- Memo 补 start → 删 Memo + 回流视图。
- 填写确定 due/end → 甘特图切换为真实截止，不再使用虚拟值。

## 7. 任务层级规则

- 支持父子关系、展开/折叠。
- 最多 3 层；禁止把子孙任务设为父任务。
- 父任务汇总规则见 5.5。

## 8. 管理员后台功能（MVP）

### 8.1 用户管理

- 用户列表、详情、启用/禁用。

### 8.2 订阅管理

- 查看订阅列表与状态；管理员手动修改有效期与 payment_status。
- MVP 不对接真实支付；不建订单/支付流水表。

## 9. MVP 范围

- 登录注册、单用户空间隔离。
- 首页三块（桌面同屏 / 移动 Tab）。
- 计划层级：goal / phase / weekly / daily。
- 任务 CRUD、父任务/子任务。
- 分流 + 备忘录（含 Memo 同步与硬删除回流）。
- 长期规划页、短期计划页。
- 后台：用户 + 订阅管理。

**不在 MVP**：复盘、真实支付、订单流水、运营配置页。

## 10. 技术选型要求

- 前端：Next.js（React + TypeScript）+ Tailwind CSS。
- 数据库：Supabase / PostgreSQL（推荐 RLS 或 middleware 强制 user_id 过滤）。
- 认证：邮箱 + 密码。
- 日历：react-big-calendar 或同类库。
- 甘特图：frappe-gantt 或同类库（需支持层级展示）。
- 路由与 API：严格按 `docs/routes.md`。
- 数据模型：严格按 `docs/data-model.md`。
- 执行顺序：严格按 `docs/plan.md`。

## 11. 验收标准

1. 用户可注册并登录，只能看到自己的数据。
2. 可创建 goal → phase → weekly/daily 计划层级。
3. 新增任务/计划后 Feed 立即有记录。
4. 无日期任务/计划只在备忘录可见，且存在对应 Memo。
5. 仅有 start 的条目：日历可见；甘特图可见，截止为 `max(start+365, today)` 虚拟值，库中 due/end 仍为空。
6. 填写 due/end 后，甘特图改用真实截止，虚拟规则停止。
7. 禁止仅填截止日期/结束日期（表单校验拦截）。
8. 虚拟截止的甘特条有视觉区分（虚线或「预估截止」）。
9. 备忘录补 start 后 Memo 被删除，内容回流日历/甘特图。
10. 编辑任务/计划标题后关联 Memo 同步更新。
11. 任务父子层级与父任务汇总正确。
12. 管理员可管理用户与订阅；普通用户不能访问 `/admin/*`。
